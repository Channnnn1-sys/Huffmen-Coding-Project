# Huffman File Compression Web Application

## Overview

This project is a web-based Huffman compression system powered by a native C++ computational engine with Flask serving as the lightweight interface and deployment layer.

The application uses compiled C++ for the real Huffman compression and decompression work, while Flask handles uploads, HTTP routing, temporary session directories, JSON responses, and browser download packaging. It is designed for Render/Linux deployment and follows a stateless cloud architecture in which uploads are processed in temporary session directories and cleaned up automatically.

TXT files are the primary effective evaluation format because Huffman coding performs best on text-like data with repeated symbols and low-to-moderate entropy. Binary formats such as PDF, DOCX, JPG, and MP4 may not compress effectively because they often already contain high entropy, internal compression, or highly random content, which limits the usefulness of prefix-free Huffman coding.

This project remains intentionally simple and academic: Flask handles HTTP, templates, routing, and ZIP packaging, while the compiled C++ engine performs the actual computational work. The Python side is kept lightweight so the system stays easy to deploy on Render/Linux and easy to explain in a research or defense setting.

## Supported File Types

### Compression

The application accepts many file types for demonstration, but the most reliable compression results are expected on TXT, CSV, LOG, JSON, XML, and source-code files.

Binary or already-compressed content such as PDF, DOCX, JPG, PNG, MP3, MP4, and ZIP may not reduce in size meaningfully because the data already has high entropy or internal compression.

Examples of practical input types include:

* TXT
* DOC
* DOCX
* PPT
* PPTX
* XLSX
* PDF
* CSV
* LOG
* JSON
* XML
* Images
* Binary files
* Source code files

Compression efficiency depends on the contents of the file.

### Decompression

The decompression page accepts:

* `.bin` files generated by this application
* ZIP packages generated by this application

For best results, users should decompress files that were originally compressed using this system.

# System Architecture

```text
Browser
   │
   ▼
Flask Application
   │
   ▼
Temporary Session Directory
   │
   ▼
C++ Huffman Compressor / Decompressor
   │
   ▼
ZIP Packaging
   │
   ▼
Base64 Encoding
   │
   ▼
JSON Response
   │
   ▼
Browser Download
```

The application uses a stateless cloud-friendly architecture suitable for Render and Linux hosting.

No uploaded files remain on the server after processing is complete, and the C++ engine remains the main computational component behind the Flask interface.

---

# Compression Workflow

1. User uploads one or more files.
2. Flask saves files into a temporary session directory.
3. File analysis is performed.
4. Entropy analysis is generated.
5. Office file deep scanning is performed when applicable.
6. The C++ Huffman compressor is executed through a subprocess.
7. Compressed output is generated.
8. Compression statistics are calculated.
9. Results are packaged into a ZIP archive.
10. ZIP archive is encoded into Base64.
11. Browser automatically downloads the result.

---

# Decompression Workflow

1. User uploads a compressed `.bin` file or generated ZIP package.
2. Flask saves the file into a temporary session directory.
3. ZIP contents are extracted when necessary.
4. The C++ Huffman decompressor is executed.
5. Original file is reconstructed.
6. Results are packaged into a ZIP archive.
7. Browser downloads the restored file.

---

# Entropy Analysis

The application performs Shannon Entropy analysis on uploaded files.

Entropy helps estimate how compressible a file is.

Examples of files with high entropy:

* PDF
* ZIP
* RAR
* JPG
* PNG
* MP4
* MP3

When entropy is high, the application warns users that additional compression may produce limited savings.

---

# Office Deep Scan

The application performs additional analysis for:

* DOCX
* PPTX
* XLSX

These file types are ZIP-based Office containers.

The system scans XML content inside:

* Word documents
* PowerPoint slides
* Excel workbooks

This analysis provides insight into how much textual content exists within the file and helps explain expected compression performance.

# Performance Notes

Compression performance depends on:

* File size

* Text files compress very well.
* Source code compresses well.
* CSV and logs compress well.

---

# Limitations

Current limitations include:

* Files are processed sequentially.
* Very large files may require longer processing times.
* Compression effectiveness varies by file type.
* Base64 packaging increases memory usage during transfer.
* Huffman Coding alone cannot outperform modern archive formats such as ZIP, 7-Zip, or RAR on already compressed files.
